Technical Overview – Hinshaw Mortgage

Project Overview

The goal of this project was to recreate the full analytics environment I worked with at Rocket Mortgage—without using any real customer or operational data. Mortgage data is highly sensitive, tightly regulated, and cannot be shared outside the company, which meant I needed a way to demonstrate my technical skills, domain knowledge, and end‑to‑end workflow without exposing proprietary information. The solution was to engineer a fully synthetic mortgage pipeline that mirrors the structure, behavior, and complexity of a real lending operation.

To accomplish this, I designed a fictional mortgage company—Hinshaw Mortgage—and generated three core datasets in Python: a loan pipeline, an extension dataset, and an employee roster. These datasets were intentionally modeled after the types of data sources I worked with in production: multi‑table relationships, operational metrics, timeline‑driven events, and performance indicators that require transformation before they become meaningful.

Once the synthetic data was created, I built a complete analytics pipeline around it—cleaning and transforming the data in Jupyter Notebook, loading it into PostgreSQL, modeling relationships between tables, and connecting the final structure to Power BI. From there, I developed more than fifty custom DAX measures to replicate the KPI logic, scoring systems, and operational insights used in real mortgage analytics. The result is a set of dashboards that behave like the tools used inside a mortgage company, but built entirely on safe, synthetic data.

This Technical Overview page documents the full engineering process behind the dashboards: how the data was generated, how the ETL workflow was structured, how the SQL model was designed, how the Power BI measures were built, and how the final project was deployed as a public portfolio site. It serves as the technical backbone of the Hinshaw Mortgage project and demonstrates the complete analytics workflow—from raw data creation to polished, interactive reporting.

1. Synthetic Data Generation (Python)

Using Jupyter Notebook as the development environment, I built a Python script that generates the synthetic mortgage datasets step‑by‑step, mirroring the structure and behavior of real production data. The script uses a combination of Pandas for dataframe construction, NumPy for statistical distributions, random for controlled randomness, and Python’s datetime library to simulate realistic timelines such as application dates, lock periods, and funding lags.

The data generation script produces three core datasets: Employees, Loans, and Loan Extensions. Each dataset is created using domain rules, weighted randomness, and capped normal distributions that reflect real mortgage operations. The script is structured in three major sections:

1. Employee Dataset — The script begins by defining a role configuration that determines how many employees exist in each position (loan officers, processors, underwriters, account executives, and leadership roles). Using loops and weighted selections, each employee is assigned:

  • A randomly selected first and last name
  • A branch and region pulled from a predefined branch list
  • A hire date generated within a realistic 10-year window
  • A compensation structure based on role (salary vs. bps)
  • A unique employee ID

A helper function retrieves employee IDs by role, allowing the loan generation process to assign each loan to a realistic team structure. This mirrors how real mortgage companies organize production teams.

2. Loan Pipeline Dataset — The script then generates 10,000 synthetic loans. Each loan is built using logic that reflects real mortgage behavior:

  • Application and lock dates are generated using date offsets from a base date to simulate a rolling pipeline.
  • Loan amounts are drawn from NumPy normal distributions that vary by state, producing realistic price differences between markets like CA, TX, and NC.
  • Product type is determined by loan amount (e.g., jumbo if above the conforming limit) or by weighted random selection.
  • Credit metrics (FICO, DTI, CLTV) use capped normal distributions to mimic real borrower profiles while enforcing realistic minimums and maximums.
  • Loan status (Funded, Withdrawn, Denied, etc.) is assigned using weighted probabilities to reflect real pull‑through behavior.
  • Employee assignments link each loan to a loan officer, processor, underwriter, and (if wholesale) an account executive.
  • Compensation is calculated using the loan amount and a randomized bps value tied to the loan officer’s comp plan.

This produces a loan pipeline that looks and behaves like the datasets used inside a real mortgage company—complete with production variation, channel differences, and realistic borrower characteristics.

3. Extension Dataset — For loans that fund, the script determines whether extensions occur based on product type, purpose, and operational randomness. Each extension record includes:

  • The number of extension days (5 or 15, depending on sequence)
  • The fee percentage and fee amount, calculated from the loan amount
  • A reason code (borrower delay, appraisal delay, underwriting conditions, etc.)
  • The request date and before/after lock expiration dates
  • The employee who approved the extension

The extension logic uses sequential loops to update expiration dates, apply fee rules, and generate realistic extension patterns. Early extensions tend to be shorter and cheaper, while later extensions carry higher cost and longer durations, mirroring real lock‑management behavior.

By combining statistical modeling, domain rules, helper functions, and controlled randomness, the final datasets behave like real mortgage data while remaining completely synthetic. This provides a safe, realistic foundation for the SQL modeling, DAX measures, and dashboards that follow.

2. ETL & Data Cleaning (Jupyter Notebook)

After generating the synthetic datasets, I used Jupyter Notebook to perform all ETL and data‑cleaning steps before loading the data into PostgreSQL. The workflow followed a traditional Extract → Transform → Load pattern using Pandas for data manipulation, NumPy for numeric operations, and SQLAlchemy with psycopg2 to load the cleaned data into a relational database.

Extract — I began by loading the raw CSV files (employees.csv, loans.csv, loan_extensions.csv) into Pandas dataframes. Using .head(), .info(), and .columns, I validated row counts, inspected data types, identified missing values, and confirmed the structure of each dataset. This step ensured that the generated data aligned with the expected schema before any transformations were applied.

Transform — Each dataset required extensive cleaning and standardization to prepare it for SQL modeling and Power BI. Key transformations included:

  • Column normalization: Standardized column names by stripping whitespace, converting to lowercase, replacing spaces with underscores, and then renaming to business‑friendly labels.
  • Date parsing: Converted all date fields (hire dates, application dates, lock dates, funding dates, extension dates) using pd.to_datetime() with error coercion.
  • Numeric conversion: Coerced salary, BPS compensation, loan amounts, interest rates, extension fees, and other numeric fields into proper numeric types.
  • Text cleaning: Standardized categorical fields such as extension reasons and requestor roles by stripping whitespace, normalizing spacing, and applying consistent capitalization.
  • Feature engineering: Added new fields including employee tenure, compensation flags, branch labels, total extension days, total extension cost, and extension buckets.
  • Duplicate checks: Verified dataset integrity using .duplicated() across full rows and key combinations (e.g., loan number + extension sequence).
  • Missing value handling: Identified and resolved null values where appropriate, ensuring all required fields were populated for SQL ingestion.

These transformations ensured that each dataset was clean, consistent, and aligned with real mortgage data structures. For example, employee compensation fields were mutually exclusive (salary vs. BPS), loan dates were properly typed for timeline calculations, and extension records included aggregated metrics such as total extension days and total extension cost per loan.

3. Loading Cleaned Data into PostgreSQL

Once the datasets were cleaned, I used SQLAlchemy and psycopg2 to load them into a PostgreSQL database named Hinshaw_Mortgage. I created three tables—employees_clean, loans_clean, and extensions_clean—using explicit CREATE TABLE statements to define data types, primary keys, and foreign key relationships.

After defining the schema, I used to_sql() to append the cleaned Pandas dataframes into their respective tables. To validate the load, I ran SQL queries directly from the notebook (e.g., SELECT COUNT(*) FROM employees) to confirm that row counts matched the cleaned datasets.

This step completed the ETL pipeline and prepared the data for SQL modeling and Power BI. By loading the data into PostgreSQL, I was able to write relational queries, enforce schema consistency, and build a more realistic analytics environment that mirrors the workflows used in enterprise BI systems.
